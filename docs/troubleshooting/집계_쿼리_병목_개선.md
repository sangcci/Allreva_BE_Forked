# 개요

현재 ALLREVA 프로젝트에서 차량 대절 신청 시 발생하는 문제와 해결 방안을 정리한다.

1. 데이터 증가에 따른 rent_participants 조회 성능 저하

soft delete를 적용하면 rent_participants(신청자) 데이터가 계속 누적된다.

boarding_date를 기준으로 WHERE 조건을 걸어 조회하지만, 데이터가 많아질수록 성능이 저하된다. boarding_date에 인덱스를 추가하면 성능을 개선할 수 있으나, 다음과 같은 한계가 있다.

- COUNT 쿼리로 인해 설계 복잡도 증가
- JOIN 과정에서 rent_participants 테이블 스캔이 사실상 필수
- 단순 COUNT가 아니라 SUM(passenger_num)을 계산해야 하므로 쿼리가 더 복잡해짐

요구사항이 변경될 때마다 쿼리가 더 복잡해져 도메인 이해도와 유지보수성이 떨어진다.

예를 들어 rent_participants를 JOIN한 뒤 SUB QUERY로 SUM을 구하는 방식은 가독성이 낮고 성능도 좋지 않다.

### 기존 QueryDSL 쿼리 예시

```java
public List<RentParticipantResponse> findRentParticipant(final Long memberId) {
    QRent rentSub = new QRent("rentSub");
    QRentBoardingSlot rentBoardingSlotSub = new QRentBoardingSlot("rentBoardingSlotSub");
    QRentParticipant rentParticipantSub = new QRentParticipant("rentParticipantSub");

    return queryFactory.select(Projections.constructor(RentParticipantResponse.class,
            rent.id,
            rent.detailInfo.title,
            rentBoardingSlot.date,
            rent.operationInfo.boardingArea,
            rent.createdAt,
            rent.additionalInfo.endDate,
            rent.additionalInfo.recruitmentCount,
            ExpressionUtils.as(
                    JPAExpressions.select(rentParticipantSub.passengerNum.sum())
                            .from(rentSub)
                            .join(rentBoardingSlotSub).on(rentSub.id.eq(rentBoardingSlotSub.rent.id))
                            .join(rentParticipantSub).on(rentBoardingSlotSub.date.eq(rentParticipantSub.boardingDate))
                            .where(rentSub.id.eq(rent.id)),
                    "passengerNum"
            ),
            rent.isClosed,
            rentParticipant.id,
            rentParticipant.createdAt,
            rentParticipant.passengerNum,
            rentParticipant.boardingType,
            rentParticipant.depositor.depositorName,
            rentParticipant.depositor.depositorTime,
            rentParticipant.refundType
    ))
            .from(rent)
            .join(rentBoardingSlot).on(rent.id.eq(rentBoardingSlot.rent.id))
            .join(rentParticipant).on(rentBoardingSlot.date.eq(rentParticipant.boardingDate))
            .where(rentParticipant.memberId.eq(memberId))
            .groupBy(rent.id, rentBoardingSlot.date, rentParticipant.id)
            .fetch();
}
```

이 방식은 복잡한 GROUP BY와 SUB QUERY를 유발해 SQL 실행 계획이 비효율적으로 동작하게 만든다.

### 대안: 신청자 수 컬럼 역정규화 적용

보다 효율적인 조회를 위해 rent_boarding_slots 테이블에 passenger_count 컬럼을 추가하여 역정규화를 적용한다.

차량 대절을 신청할 때 rent_participants 테이블에 레코드를 추가하기 전, passenger_count 값을 증가시키는 방식.

기존 방식 vs. 개선 방식

- 기존: SUM(passenger_num)을 rent_participants에서 계산
- 개선: passenger_count 컬럼에서 직접 조회

쿼리 복잡도가 줄고 단순 조회로 성능이 크게 향상된다.

### 추가 효과:

기존 검증 로직:

```java
private void checkPassengersMaximumReached(
				final Long rentId,
				final LocalDate boardingDate,
				final Integer passengerNum
) {
		Rent rent = rentRepository.findById(rentId)
						.orElseThrow(RentNotFoundException::new); // 1. rent 조회 쿼리
		int maximumCount = rent.getAdditionalInfo().getRecruitmentCount();
		
		Integer currentPassengerCount = rentParticipantRepository.countRentParticipants(rentId, boardingDate); // 2. 집계 함수 쿼리
		if (currentPassengerCount + passengerNum > maximumCount) {
		    throw new PassengersMaximumReachedException();
		}
		// rent close event
		if (currentPassengerCount + passengerNum == maximumCount) {
		    Events.raise(new RentClosedEvent(rentId));
		}
}
```

- rent에서 모집 인원 조회 후 rent_participants에서 Aggregation으로 신청 인원 계산 → 복잡

개선 로직:

```java
private void checkPassengersMaximumReached(
				final Long rentId,
				final LocalDate boardingDate,
				final Integer passengerNum
) {
		RentBoardingSlot rentBoardingSlot = rentRepository.findRentBoardingSlot(rentId, boardingDate)
		.orElseThrow(() -> new RentNotFoundException(rentId));
		int maximumCount = rentBoardingSlot.getRecruitmentCount();
		int currentPassengerCount = rentBoardingSlot.getPassengerCount();

		}
}
```

- 비즈니스 로직 집중
- 🤔 동시성 이슈는 별도 해결 필요

이외에도 QueryDSL을 이용해 쿼리 코드를 작성하는 과정에서도 복잡성을 대폭 줄일 수 있었다.

### 결론

passenger_count 컬럼을 rent_boarding_slots 테이블에 추가해 역정규화하면 쿼리가 단순해지고 성능이 크게 개선된다.

개선 효과

- Aggregation → 인덱스 추가 시 성능 차이 미미
- SUB QUERY 제거 → 조회 성능 향상
- QueryDSL 쿼리 단순화 → 유지보수성 ↑

1. 느낀 점

- Aggregation Query는 신중히 도입
- 데이터 모델링 시 count 데이터는 단일 컬럼으로 관리 고려
- 동시성 등 부수적 문제는 추가 공부 필요

1단계 - Bitmap Index Scan

member_id = 1에 해당하는 row들의 물리적 위치(페이지 번호)를 비트맵으로 수집.

페이지 번호 : 0 1 2 3 4 5 6 7 … 비트맵: 0 1 0 0 1 0 1 0 …

비트맵: bit를 페이지 번호에 대응시킨 배열.

페이지 안에 여러 튜플이 저장되어 있어 같은 페이지를 접근해야 하는 상황 발생. 이럴 때 랜덤 I/O 낭비를 줄이기 위해 비트맵으로 수집한 후 Bitmap Heap Scan 시 비트맵을 보고 실제 Heap 테이블을 읽음.

즉 스캔이 한 번 전체적으로 이루어진 후 힙 영역을 순차로 또 스캔함(이때는 실제 페이지만 순차로 읽음)

즉 member_id=1 데이터들이 물리적으로 많이 분산되어 있다는 소리…

heap sort는 최소 힙 구조임. 결국 30,000개를 전체를 순회해서 가장 우선순위가 최소인 값을 계속 탈락시키는 구조임. 그러면 자연스럽게 우선순위가 높은 10개만 남게되는 구조. 이게 O(n log 10)이기 때문에 정말 쌈. 힙 재구성 비용이 log 10이라서 사실상 상수 수준. 이 힙 재구성을 30,000번 반복하면 되는데 이게 정말 쌈.

문제는 비트맵 스캔임. 인덱스가 있어도 결과 값이 30,000개면 얘기가 달라짐. 랜덤 I/O를 30,000번 수행해야 하기 때문에 성능 제약이 생김. 아무리 Bitmap으로 랜덤 I/O를 줄이려 노력해도 무시할 수 없는 수준임.
