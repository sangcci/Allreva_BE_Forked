# 차량 대절 참여 내역 조회 인덱스 최적화 — 70.69ms → 1.02ms 개선

## 전체 흐름

```text
[Before]
member_id 조건으로 참여 내역 조회
  → member_id = 1 후보 30,000건 탐색
  → Bitmap Heap Scan
  → Sort(boarding_date DESC, id DESC)
  → LIMIT 10
  = 70.69ms

[After]
(member_id, boarding_date, id) 복합 인덱스 적용
  → 조건과 정렬 순서를 인덱스에서 함께 처리
  → Index Scan Backward
  → LIMIT 10
  = 1.02ms
```

## 문제 상황

차량 대절 참여 내역 조회는 무한 스크롤 방식으로 최신 참여 내역 10건만 내려주면 되는 API였습니다. 그런데 실제 실행 계획은 최신 10건만 바로 가져오는 구조가 아니었습니다.

테스트에서는 사용자 1명이 차량 대절에 30,000건 참여한 상황을 가정했습니다. 캐시 영향을 줄이기 위해 매 시도마다 PostgreSQL을 재시작해 `shared_buffers`를 비운 뒤 실행 계획을 확인했습니다.

조회 쿼리는 `member_id`로 참여 내역을 제한하고, `(boarding_date, id)` 커서를 기준으로 다음 데이터를 조회한 뒤 최신순으로 10건만 반환하는 구조였습니다.

```sql
EXPLAIN (ANALYZE, BUFFERS)
SELECT rp.boarding_date, rp.rent_id, rp.member_id, rp.id, rp.passenger_num
FROM rent_participants AS rp
WHERE rp.member_id = 1
  AND (boarding_date, id) < ('2025-09-21', 925107)
ORDER BY rp.boarding_date DESC, rp.id DESC
LIMIT 10;
```

## 원인 분석

기존 인덱스는 `member_id` 조건으로 후보를 줄이는 데는 일부 도움을 줬지만, `ORDER BY boarding_date DESC, id DESC` 정렬 순서까지 만족하지 못했습니다.

그 결과 PostgreSQL은 아래 순서로 동작했습니다.

1. `member_id = 1` 조건에 맞는 후보 30,000건을 찾음
2. 커서 조건에 맞지 않는 일부 row를 필터링
3. heap block을 넓게 읽음
4. `boarding_date DESC, id DESC` 기준으로 다시 정렬
5. 마지막에 10건만 반환

즉 `LIMIT 10`이 있어도 앞단에서 너무 많은 데이터를 읽고 정렬한 뒤 버리는 구조였습니다.

## 해결 과정

해결 방향은 단순히 인덱스를 하나 추가하는 것이 아니라, 조회 조건과 정렬 조건을 함께 만족하는 접근 경로를 만드는 것이었습니다.

그래서 `rent_participants`에 아래 복합 인덱스를 적용했습니다.

```sql
CREATE INDEX idx_rent_participants
ON rent_participants (member_id, boarding_date, id);
```

이 인덱스로 기대한 효과는 두 가지였습니다.

- `member_id = 1` 조건으로 후보 범위를 빠르게 제한
- `(boarding_date, id)` 기준 커서 조건과 정렬 순서를 인덱스 스캔 단계에서 함께 처리

무한 스크롤 조회는 페이지 번호 기반 `OFFSET`보다 커서 기반 조회가 더 잘 맞았습니다. 사용자는 “정확한 10페이지”보다 “이전 목록 다음 10개”를 필요로 하기 때문입니다. 그래서 쿼리 구조도 `(boarding_date, id)` 커서를 기준으로 다음 데이터를 가져오도록 정리했습니다.

## 결과

### After 실행 계획

```text
Limit  (cost=0.42..27.14 rows=10 width=32) (actual time=0.312..0.974 rows=10 loops=1)
  Buffers: shared read=13
  ->  Index Scan Backward using idx_rent_participants on rent_participants rp  (cost=0.42..54450.12 rows=20383 width=32) (actual time=0.308..0.967 rows=10 loops=1)
        Index Cond: ((member_id = 1) AND (ROW(boarding_date, id) < ROW('2025-09-21'::date, 925107)))
        Buffers: shared read=13
Planning Time: 1.942 ms
Execution Time: 1.020 ms
```

실행 계획은 훨씬 단순해졌습니다.

- `Bitmap Heap Scan` 제거
- `Sort` 제거
- `Index Scan Backward`로 필요한 순서대로 10건만 조회
- buffer read `19643` → `13`
- 실행 시간 `70.69ms` → `1.02ms`

## 정리

이번 개선의 핵심은 “인덱스를 추가했다”가 아니라, API 조회 방식에 맞춰 쿼리와 인덱스를 함께 설계했다는 점입니다.

무한 스크롤에서는 커서 기준값을 활용할 수 있고, 이 커서 조건과 정렬 순서를 복합 인덱스에 맞추면 DB가 불필요한 후보 row를 읽고 정렬하는 비용을 줄일 수 있습니다. 결과적으로 최신 10건만 필요한 API가 실제로도 10건에 가까운 범위만 읽도록 바뀌었습니다.

## 남은 과제

현재 실행 계획은 `Index Only Scan`이 아니라 `Index Scan`입니다. 따라서 heap 접근이 이론적으로 완전히 사라졌다고 볼 수는 없습니다. 다만 이전처럼 대량 후보를 모으고 정렬하던 구조는 제거됐습니다.

추가로 SELECT 컬럼과 visibility map 상태에 따라 `Index Only Scan` 가능성을 검토하거나, 실제 운영 데이터 분포에서 같은 실행 계획이 유지되는지도 계속 확인할 필요가 있습니다.
