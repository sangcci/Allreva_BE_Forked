## 추가 설계 개선 사항 (2026-04-30)

### 1. Search 모듈 분리 필요성

PostgreSQL trgm 기반에서는 검색이 단순 WHERE 쿼리라 도메인 모듈에 붙이는 게 맞음.
별도 search 모듈이 의미 있는 조건은 Elasticsearch처럼 별도 인덱스 관리가 필요한 경우.
인기 검색어(`/search/popular`)처럼 cross-domain인 경우만 search 모듈에 유지.

### 2. PlaceSearchService 위치 문제

`PlaceSearchService`가 search 모듈 안에 있지만 `concert.place` DTO와 ErrorCode를 직접 참조.
사실상 concert.place 도메인 서비스이므로 `concert.place` 모듈로 이동이 적절.
`SearchController`는 search 모듈이 domain에 의존하는 설계가 허용되므로 concert.place 서비스를 직접 호출하면 됨.

### 3. 페이지네이션 래퍼 DTO 중복

동일한 구조의 래퍼 DTO가 도메인마다 중복 정의되어 있음:
- `ConcertSearchListResponse` — `List<ConcertThumbnail>` + `nextCursorCode`
- `RentSearchListResponse` — `List<RentThumbnail>` + `nextCursorId`
- `SurveySearchListResponse` — `List<SurveyThumbnail>` + `nextCursorId`

**해결:** `common` 패키지에 `SliceResponse<T>` 제네릭 래퍼 도입.

```java
public record SliceResponse<T>(
    List<T> items,
    Object nextCursor
) {}
```

cursor 타입이 `String`(hallCode), `Long`(id) 등 도메인마다 다르므로 `Object`로 통일.
타입 안전이 필요하면 `SliceResponse<T, C extends Comparable<C>>` 형태도 가능하나 사용처가 장황해짐.

### 4. hallCode cursor paging 적합성

hallCode 패턴: `FC000001-01` (0-padded 숫자 코드)
사전순 정렬 = 숫자 순서와 일치하므로 cursor paging 조건 만족.
`WHERE hall_code > :lastHallCode` 방식으로 커서 적용 가능.
