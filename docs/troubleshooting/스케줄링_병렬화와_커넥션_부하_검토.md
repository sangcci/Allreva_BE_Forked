# 개요

Concert / ConcertHall 동기화 스케줄을 보면서, 스케줄링 작업을 더 빠르게 끝내기 위해 내부 병렬화를 도입해도 되는지 검토했다. 처음에는 레코드 식별자만 잘 나누면 충분하다고 생각했다. 하지만 실제로는 단순히 레코드 충돌만 피하면 끝나는 문제가 아니었다.

지금 운영 환경에서는 WAS 인스턴스가 1개다. 이 조건이 생각보다 중요했다. 스케줄러가 병렬로 동작하면 같은 시점에 더 많은 DB 커넥션을 점유할 수 있고, 그만큼 사용자 요청이 사용할 수 있는 커넥션이 줄어든다. 결국 이번 고민의 핵심은 "병렬화가 가능한가"보다 "지금 운영 조건에서 병렬화가 정말 이득인가"에 더 가깝다고 봤다.

# 현재 전제

- Spring Boot 3.3
- JDK 17
- 단일 WAS 인스턴스 운영
- KOPIS 연동 스케줄 존재
- DB 커넥션은 HikariCP 풀을 통해 사용
- 현재 스케줄 작업은 배치 프레임워크 기반이 아니라, 스케줄러 메서드 안에서 순차 루프로 처리하는 구조

# 처음 들었던 생각

처음에는 이렇게 생각했다.

- 스케줄러가 내부에서 병렬로 돌더라도
- hallCode, facilityCode, concertCode처럼 식별자만 잘 나누고
- 서로 같은 레코드를 동시에 건드리지 않게 하면
- 더 빨리 끝낼 수 있지 않을까

이 생각 자체는 완전히 틀린 건 아니다. 실제로 작업 단위를 잘 나누면 병렬화로 전체 시간을 줄일 수 있다. 다만 운영에서는 그보다 먼저 봐야 할 게 더 많다.

# 왜 단순히 레코드 충돌 문제로만 보면 안 되는가

## 1. 스케줄러 스레드와 DB 커넥션은 다른 레이어다

스케줄러는 스레드 위에서 동작하고, DB 쓰기는 트랜잭션과 JDBC 커넥션 위에서 수행된다. 그래서 병렬화를 검토할 때는 "스레드가 몇 개 필요한가"만 보면 부족하다.

실제로는 아래를 같이 봐야 한다.

- 동시에 몇 개의 작업이 실행되는가
- 각 작업이 트랜잭션을 얼마나 오래 유지하는가
- 각 작업이 HikariCP 커넥션을 얼마나 오래 점유하는가
- 같은 시점에 사용자 요청도 커넥션을 필요로 하는가

즉, 스레드 수를 늘리는 결정은 결국 커넥션 사용량과 바로 연결된다.

## 2. 단일 WAS 환경에서는 커넥션 경쟁이 더 직접적이다

지금은 WAS가 1대뿐이다. 그래서 스케줄러가 커넥션을 많이 점유하면, 그 영향이 다른 인스턴스로 분산되지 않는다. 사용자가 들어오는 요청도 같은 HikariCP 풀을 쓴다.

이 상황에서는 스케줄 작업을 병렬화할수록 다음 문제가 생길 수 있다.

- 사용자 요청이 커넥션을 대기한다
- 응답 시간이 늘어난다
- 피크 타이밍에는 타임아웃 가능성도 생긴다

그래서 이번 고민에서 결정적인 포인트는 이것이었다. 지금 구조에서는 병렬화의 이득보다, 사용자 요청 커넥션을 잠식하는 비용이 더 클 수 있다.

## 3. JPA 기반 병렬화는 생각보다 조심해야 한다

현재 구조처럼 JPA Repository를 통해 저장하는 코드에서는, 메서드 안에서 단순히 `parallelStream()` 을 적용하는 식의 접근은 위험할 수 있다.

이유는 다음과 같다.

- 영속성 컨텍스트는 스레드 안전하지 않다
- 트랜잭션 경계가 흐려질 수 있다
- flush / save 시점이 예상과 다르게 꼬일 수 있다
- 디버깅 난도가 급격히 올라간다

병렬화를 하더라도 엔티티를 공유하지 않고, 식별자나 DTO 중심으로 작업을 분리한 뒤 각 작업이 독립 트랜잭션을 갖도록 설계하는 편이 안전하다.

## 4. 외부 API rate limit도 무시할 수 없다

이 문제는 DB만의 문제가 아니다. KOPIS 연동처럼 외부 API를 호출하는 구조에서는 병렬화가 곧바로 호출량 증가로 이어진다.

즉 병렬화 이후에는 아래를 같이 봐야 한다.

- 외부 API 호출 제한
- 실패율 증가 여부
- 재시도 로직 유무
- 장애 시 전체 작업이 얼마나 흔들리는지

병렬화로 로컬 처리 시간은 줄었는데, 외부 API throttling 때문에 전체 작업은 오히려 더 느려질 수도 있다.

# 그럼 스케줄링에 여러 스레드가 꼭 필요한가

현재 구조만 놓고 보면, 각 스케줄 작업 내부에 여러 스레드가 꼭 필요하지는 않다.

- `concert` sync 하나가 돌 때 스레드 1개
- `concert hall` sync 하나가 돌 때 스레드 1개

정도면 기능적으로는 충분하다.

다만 이것과 별개로, 스케줄러 풀 크기는 "작업 내부 병렬성"이 아니라 "동시에 실행되어야 하는 스케줄 작업 개수" 관점에서 봐야 한다.

예를 들어:

- 스케줄 작업이 하나뿐이면 pool size 1로도 충분할 수 있다
- 서로 다른 스케줄이 겹쳐도 독립적으로 돌아야 하면 2 이상이 필요할 수 있다

즉, 내부 병렬화가 필요 없다는 말과 scheduler pool은 1이면 된다는 말은 완전히 같은 얘기는 아니다.

# 지금 시점에서의 판단

지금 운영 조건에서는 병렬화를 서둘러 넣기보다, 먼저 실제 운영 지표를 보고 판단하는 편이 맞다고 생각한다.

그 이유는 다음과 같다.

1. 단일 WAS라서 커넥션 경쟁이 직접적이다
2. 스케줄 작업이 사용자 요청과 같은 커넥션 풀을 사용한다
3. 외부 API 호출 제한도 함께 고려해야 한다
4. JPA 기반 병렬화는 설계를 잘못하면 복잡성만 크게 늘어난다

그래서 현재로서는 "병렬화가 가능하다"는 결론보다, "운영에서 병목이 실제로 확인되면 그때 안전한 단위로 병렬화하자"는 쪽이 더 현실적이다.

# 운영에서 같이 봐야 할 항목

병렬화 여부를 결정하기 전에 아래 항목을 먼저 확인하는 것이 좋다.

## 애플리케이션 레벨

- 스케줄 시작 / 종료 / 실패 로그
- 실행 시간(duration)
- 처리 건수
- 실패한 hall / facility 수
- 스케줄이 서로 겹치는지 여부

## DB / 커넥션 풀 레벨

- HikariCP active connection 수
- idle connection 수
- connection acquire 대기 시간
- 사용자 요청 피크 시간대와 스케줄 실행 시간대의 겹침 여부
- DB slow query 발생 여부

## 외부 API 레벨

- KOPIS 호출 횟수
- 실패율
- 재시도 횟수
- rate limit 징후

## 사용자 체감 레벨

- 스케줄 실행 시간대의 API 응답 시간 증가 여부
- 타임아웃 / 5xx 발생 여부

# 병렬화를 한다면 어떻게 접근할 것인가

바로 `parallelStream()` 을 넣기보다는, 아래처럼 점진적으로 보는 편이 안전하다.

1. 먼저 현재 구조에서 실행 시간과 실패율을 관측한다
2. 병목이 분명할 때만 병렬화 후보를 정한다
3. 병렬화 단위는 hallCode / facilityCode처럼 충돌 범위가 작은 단위로 자른다
4. 엔티티 공유 대신 식별자 기반으로 작업을 분리한다
5. 각 작업은 짧은 트랜잭션으로 처리한다
6. HikariCP 풀과 사용자 요청 부하를 같이 보면서 동시성을 낮게 시작한다

즉 병렬화는 가능하지만, 지금은 "빠를 것 같다"는 감으로 넣기보다 "운영에서 이 정도면 감당 가능하다"는 근거를 확보한 뒤에 가는 게 맞다.

# 결론

이번 고민을 통해 정리한 결론은 이렇다.

스케줄링 내부 병렬화는 기술적으로 가능하다. 식별자를 잘 나누면 충돌도 어느 정도 통제할 수 있다. 하지만 지금 운영 환경은 WAS가 1개뿐이고, 사용자 요청과 스케줄 작업이 같은 커넥션 풀을 공유한다. 이 조건에서는 병렬화가 작업 시간을 줄이는 대신, 사용자 요청이 사용할 커넥션을 부족하게 만들 가능성이 더 결정적인 이슈가 된다.

그래서 현재로서는 병렬화 자체를 바로 도입하기보다, 운영에서 스케줄 실행 시간, HikariCP 상태, 외부 API 호출 실패율, 사용자 요청 응답 지표를 함께 관찰한 뒤 판단하는 접근이 더 맞다고 본다. 필요하다면 그때 hall / facility 단위의 제한된 병렬화부터 검토하는 것이 좋다.
