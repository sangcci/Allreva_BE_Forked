# 공연 조회 API 로컬 캐시 최적화 — 150ms → 5ms 개선

## 전체 흐름

```text
[Before]
Client
  → Spring Boot API
  → PostgreSQL
      pg_trgm similarity
      + ConcertHall JOIN
  = 132~150ms

[After]
Client
  → Spring Boot API
  → Caffeine Local Cache
      hit  → 즉시 반환
      miss → PostgreSQL 조회 후 캐시 적재
  = 3~18ms
```

## 문제 상황

공연 데이터는 하루 1번, 공연장 데이터는 한 달에 1번 정도만 갱신됩니다. 반면 메인 화면과 검색 API는 사용자가 자주 호출하는 조회 API입니다.

문제는 이런 데이터 특성을 활용하지 못하고, 매 요청마다 PostgreSQL에서 `pg_trgm` similarity 연산과 `ConcertHall` JOIN을 반복하고 있었다는 점입니다.

측정 당시 응답 시간은 아래 수준이었습니다.

```text
GET /api/v1/concerts/main   → 132ms
GET /api/v1/concerts/search → 150ms
GET /api/v1/concerts        → 23ms
```

기능은 동작했지만, 읽기 빈도가 높은 API에서 매번 같은 연산을 반복하는 구조라 DB 부하와 응답 시간이 모두 아쉬웠습니다.

## 캐시 대상 선정

캐시 적용 전 먼저 어떤 도메인에 로컬 캐시를 써도 되는지 구분했습니다.

| 대상 | 적용 여부 | 이유 |
|------|-----------|------|
| Concert 조회(main, search, related) | O | 일 1회 KOPIS sync, 읽기 편향 |
| ConcertHall 조회 | O | 월 1회 sync, 변경 빈도 낮음 |
| Rent | X | 좌석 수, 마감 여부 실시간성 필요 |
| Survey | X | 참가자 수가 모집 전환 조건에 직결 |

`Rent`, `Survey`는 참가자 수와 마감 여부가 중요한 도메인입니다. 멀티 인스턴스 환경에서 각 인스턴스가 다른 로컬 캐시를 들고 있으면 정합성 문제가 생길 수 있다고 판단해 제외했습니다.

반대로 `Concert`, `ConcertHall`은 변경 주기가 길고 갱신 시점이 명확했습니다. 로컬 캐시를 적용해도 정합성 리스크가 낮고, 반복 조회 비용을 줄이는 효과가 크다고 봤습니다.

## 해결 과정

캐시는 Caffeine 로컬 캐시를 사용했습니다. API 성격에 따라 TTL과 `maxSize`를 다르게 설정했습니다.

- 메인 공연 조회: 하루 단위 갱신 → TTL 1일
- 검색 조회: 검색어 조합 증가 가능 → TTL 1시간, `maxSize` 제한
- 공연장 조회: 월 단위 갱신 → TTL 30일

```java
@EnableCaching
@Configuration
public class CacheConfig {

    @Bean
    public CacheManager cacheManager() {
        SimpleCacheManager manager = new SimpleCacheManager();
        manager.setCaches(List.of(
                build("concertMain",       100, Duration.ofDays(1)),
                build("concertTop10",      10,  Duration.ofDays(1)),
                build("concertSearch",     500, Duration.ofHours(1)),
                build("concertRelated",    200, Duration.ofHours(1)),
                build("concertHall",       200, Duration.ofDays(30)),
                build("concertHallSearch", 500, Duration.ofHours(1))
        ));
        return manager;
    }

    private CaffeineCache build(String name, int maxSize, Duration ttl) {
        return new CaffeineCache(name, Caffeine.newBuilder()
                .maximumSize(maxSize)
                .expireAfterWrite(ttl)
                .build());
    }
}
```

`CaffeineCacheManager` 대신 `SimpleCacheManager`를 사용했습니다. 캐시마다 TTL과 크기를 다르게 주려면 개별 `CaffeineCache` 인스턴스를 구성하는 편이 더 명확했기 때문입니다.

## 캐시 무효화 전략

TTL은 안전장치로 두고, 실제 데이터 변경 시점에는 `@CacheEvict`로 명시적으로 무효화했습니다.

| 무효화 시점 | 대상 캐시 | 방법 |
|------------|----------|------|
| 매일 새벽 4시 KOPIS Concert sync | concertMain, concertTop10, concertSearch, concertRelated | `@CacheEvict(allEntries=true)` |
| 매월 1일 새벽 2시 KOPIS ConcertHall sync | concertHall, concertHallSearch | `@CacheEvict(allEntries=true)` |
| Admin insert/update | 각 도메인 캐시 전체 | `@CacheEvict(allEntries=true)` |

스케줄러는 같은 빈 내부에서 메서드를 호출하는 self-call 문제가 생길 수 있습니다. 그래서 스케줄러 진입 메서드와 테스트 컨트롤러에서 직접 호출하는 내부 메서드 양쪽에 `@CacheEvict`를 붙여 무효화 경로가 누락되지 않도록 했습니다.

## 결과

서버의 커스텀 MDC 로그 처리 시간을 기준으로 측정했습니다.

- PostgreSQL buffer pool warm-up 상태
- 홈서버 환경

| 엔드포인트 | 캐시 미스 | 캐시 히트 | 개선율 |
|---|---:|---:|---:|
| `GET /api/v1/concerts/main` | 132ms | 3~9ms | 약 30배 |
| `GET /api/v1/concerts/search` | 150ms | 5~6ms | 약 27배 |
| `GET /api/v1/concerts` (related) | 23ms | 3~15ms | 약 6배 |
| `GET /api/v1/concerts/{concertCode}` | 92ms | 15~18ms | 약 5배 |

메인 조회와 검색 조회에서 개선폭이 가장 컸습니다. 두 API는 기존에 `pg_trgm` similarity 연산과 JOIN 비용이 있었고, 캐시 히트 시 이 연산을 건너뛸 수 있었기 때문입니다.

단건 조회는 원래 PK 기반 조회라 절대적인 개선폭은 작았습니다. 그래도 반복 호출 시 DB 부하를 줄이는 의미는 있었습니다.

## 정리

이번 개선의 핵심은 “캐시를 붙였다”가 아니라, 데이터의 변경 주기와 조회 패턴을 먼저 보고 캐시 적용 대상을 골랐다는 점입니다.

정합성이 중요한 `Rent`, `Survey`에는 로컬 캐시를 적용하지 않았고, 변경 주기가 길고 읽기 편향이 강한 `Concert`, `ConcertHall`에만 적용했습니다. 덕분에 성능 개선과 정합성 리스크 관리를 함께 가져갈 수 있었습니다.

## 남은 과제

이번 검증은 캐시 히트/미스 응답 시간 차이에 초점을 맞췄습니다. 이후에는 실제 부하 테스트로 캐시 hit ratio, heap 메모리 사용량, eviction 패턴을 함께 확인할 필요가 있습니다.

트래픽이 늘거나 인스턴스가 여러 대로 확장된다면 로컬 캐시만으로 충분한지, Redis 같은 글로벌 캐시나 다단계 캐시 구조가 필요한지도 다시 검토해야 합니다.
