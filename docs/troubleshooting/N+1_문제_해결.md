public List<JoinedRentResponse> getJoinedRentSummaries(final Long memberId, final Long lastId, final int pageSize) {
    List<RentParticipant> participants = rentParticipantRepository.findAllByMemberId(memberId, lastId, pageSize);

    return participants.stream()
            .map(participant -> {
                 Rent rent = participant.getRent();
                 RentBoardingSlot slot = rent.getBoardingSlots().stream()
                        .filter(s -> s.getDate().equals(participant.getBoardingDate()))
                        .findFirst()
                         .orElse(null);
                return JoinedRentResponse.from(participant, rent, slot);
            })
            .toList();
}

# N+1 문제 정리
## 배경 지식
- Hibernate는 왜 LIMIT 절이 있음에도 모두 다 조회해서 Application 단에서 filtering을 진행할까?
- N+1이 일어나는 원인은?
- fetch join은 어떨 때 사용하는것인가? 
## 문제 발생

getJoinedRentSummaries()에서 participant 목록을 조회한 뒤, 각 rent의 boardingSlots 컬렉션을 순회하면서 날짜가 일치하는 slot을 찾는 과정에서 발생.

```
// participant마다 실행됨
rent.getBoardingSlots().stream()
      .filter(s -> s.getDate().equals(participant.getBoardingDate()))
      .findFirst()

-- 1번: participant + rent fetch join
SELECT rp.*, r.* FROM rent_participants rp JOIN rent r ... LIMIT 10

-- N번: 고유한 rent 수만큼
SELECT * FROM rent_boarding_slots WHERE rent_id = 1
SELECT * FROM rent_boarding_slots WHERE rent_id = 2
SELECT * FROM rent_boarding_slots WHERE rent_id = 3
...
```

## 원인 분석

직접 원인: @OneToMany는 기본 LAZY라서 getBoardingSlots() 호출 시점에 쿼리가 나간다.
근본 원인: Object-Relational Impedance Mismatch.
- 객체 모델은 `Rent`가 `List<RentBoardingSlot>`을 소유하는 구조
- 관계 모델은 JOIN 시 행이 뻥튀기되는 구조

이 간극을 메우기 위해 Hibernate가 컬렉션을 재조립하는 과정에서 문제가 생긴다.

컬렉션 fetch join + limit을 같이 쓰면 더 나빠지는 이유: JOIN으로 뻥튀기된 행 기준으로는 "participant 몇 개"인지 알 수 없어서 Hibernate가 LIMIT을 SQL에 붙이지 못하고, 전체를 가져온 뒤 Java 메모리에서 페이징한다. (HHH90003004 경고)

## 해결 방법

### 방법 1: @BatchSize (권장 - 변경 최소)

```java
@BatchSize(size = 100)
@OneToMany(mappedBy = "rent", cascade = CascadeType.ALL, orphanRemoval = true)
private List<RentBoardingSlot> boardingSlots = new ArrayList<>();
```

- 컬렉션을 별도 IN 쿼리로 분리해서 처리
- LIMIT은 1번 쿼리에서 DB 레벨로 정상 동작
- 연관관계 메서드 그대로 유지 가능
- 단, slot 날짜 필터링은 여전히 in-memory (slot 최대 10개라 무시할 수준)

```sql
-- 1번: LIMIT 정상 동작
SELECT rp.*, r.* FROM rent_participants rp JOIN rent r ... LIMIT 10

-- 1번: IN 쿼리로 묶음
SELECT * FROM rent_boarding_slots WHERE rent_id IN (1, 2, 3, 4)
```

### 방법 2: Tuple JOIN (변경 큼, 성능 최적화)

```java
queryFactory
    .select(rentParticipant, rent, rentBoardingSlot)
    .from(rentParticipant)
    .join(rentParticipant.rent, rent)
    .leftJoin(rentBoardingSlot)
        .on(rentBoardingSlot.rent.eq(rent)
            .and(rentBoardingSlot.date.eq(rentParticipant.boardingDate)))
    .where(...)
    .orderBy(rentParticipant.id.desc())
    .limit(pageSize)
    .fetch();
```

- 쿼리 1번으로 모든 것을 해결
- 날짜 조건도 DB에서 처리
- 연관관계 메서드 사용 불가, Repository/Service 시그니처 변경 필요

## 비교

```
  ┌───────────────────┬──────┬────────────┬────────────┐
  │                   │ 현재 │ @BatchSize │ Tuple JOIN │
  ├───────────────────┼──────┼────────────┼────────────┤
  │ 쿼리 수           │ 1+N  │ 1+1        │ 1          │
  ├───────────────────┼──────┼────────────┼────────────┤
  │ LIMIT DB 처리     │ O    │ O          │ O          │
  ├───────────────────┼──────┼────────────┼────────────┤
  │ 날짜 조건 DB 처리 │ X    │ X          │ O          │
  ├───────────────────┼──────┼────────────┼────────────┤
  │ 연관관계 유지     │ O    │ O          │ X          │
  ├───────────────────┼──────┼────────────┼────────────┤
  │ 코드 변경 범위    │ -    │ 최소       │ 큼         │
  └───────────────────┴──────┴────────────┴────────────┘
```

slot이 최대 10개인 이 케이스에서는 @BatchSize가 가장 현실적인 선택이다.
