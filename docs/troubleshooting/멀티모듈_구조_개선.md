# 멀티모듈 구조 개선

## 문제 상황

프로젝트가 단순 CRUD를 넘어 차량 대절, 수요 조사, 공연 동기화, 알림, 검색까지 확장되면서 Service 코드에 업무 흐름과 기술 구현이 함께 모일 가능성이 커졌습니다.

예를 들어 차량 대절 신청 흐름에는 모집 조회, 신청 가능 여부 판단, 탑승 슬롯 인원 증가, 참가자 저장, 이벤트 발행이 함께 들어갑니다. 이 절차가 하나의 Service 메서드에 길게 쌓이면 “대절 신청”이 어떤 단계로 진행되는지 읽기 어려워집니다.

또 DB 조회 방식, Redis 저장 방식, FCM 전송 방식, KOPIS 연동 방식이 usecase 안으로 들어오면 기술 변경이 비즈니스 코드 변경으로 이어질 수 있습니다.

## 원인 분석

기존 계층형 구조만으로는 두 경계가 흐려질 수 있었습니다.

1. usecase 흐름과 세부 행위의 경계
2. core 비즈니스 코드와 외부 기술 구현체의 경계

Controller-Service-Repository 구조는 익숙하지만, Service가 조회, 검증, 생성, 저장, 외부 연동 호출까지 모두 들고 있으면 코드가 절차 중심으로 길어집니다. 또한 core가 JPA, Redis, FCM, OAuth, KOPIS 같은 구현체를 직접 알면 변경 영향 범위가 커집니다.

## 해결 과정

개선 목표는 멀티모듈 자체가 아니라, 비즈니스 흐름을 읽기 쉽게 만들고 구현체를 core 밖으로 격리하는 것이었습니다.

### 1. application과 implementation 분리

`core` 내부에서 usecase 시작점은 `application` Service가 맡고, 세부 행위는 `implementation` layer로 분리했습니다.

```text
recruitment/rent
├── command/application/RentService.java
├── command/implementation/RentReader.java
├── command/implementation/RentWriter.java
├── command/implementation/RentRegister.java
├── command/implementation/RentJoiner.java
├── command/implementation/RentParticipantCanceller.java
├── command/input/RentRegisterCommand.java
├── command/event/RentRegisteredEvent.java
└── domain/Rent.java, RentParticipant.java, RentRepository.java
```

`RentService`는 usecase 순서를 보여주고, `RentReader`, `RentRegister`, `RentJoiner`, `RentWriter`는 각각 조회, 등록 객체 생성, 참여 처리, 저장을 담당합니다.

```java
@Transactional
public Long register(final RentRegisterCommand command, final Long memberId) {
    Rent rent = rentRegister.register(command, memberId);
    Rent saved = rentWriter.save(rent);
    Events.raise(RentRegisteredEvent.from(saved));
    return saved.getId();
}
```

이렇게 나누면 Service는 “등록 객체 생성 → 저장 → 이벤트 발행”이라는 흐름에 집중하고, 세부 절차는 이름 있는 행위자로 내려갑니다. 상태 변경과 불변식은 `Rent.validateMine()`, `Rent.close()`처럼 domain 객체가 맡게 했습니다.

조회는 command와 성격이 다르다고 보고 `query/application`, `query/implementation`, `query/model`로 분리했습니다. 화면/API 요구사항에 맞춘 조회는 aggregate를 억지로 재사용하지 않고 query model과 `FinderPort`로 다루었습니다.

### 2. core와 support 분리

`core`는 usecase, domain, port interface만 알고, 기술 구현체는 `support`에 둔 구조로 정리했습니다.

```text
                 ┌──────────────────┐
                 │   allreva-api    │
                 │ REST, SSE, Auth  │
                 └────────┬─────────┘
                          │
                          ▼
                 ┌──────────────────┐
                 │  allreva-core    │
                 │ usecase, domain  │
                 └────────▲─────────┘
                          │
        implements ports  │  depends on port interfaces
                          │
┌─────────────────────────┴─────────────────────────┐
│                    allreva-support                 │
│ db, redis, s3, oauth, kopis, fcm, cache, logging   │
└───────────────────────────────────────────────────┘
```

```text
allreva-support
├── db
│   ├── *Entity.java
│   ├── *JpaRepository.java
│   ├── *RepositoryImpl.java
│   └── *FinderAdapter.java
├── global-cache
├── local-cache
├── storage
├── oauth-client
├── kopis-client
└── push-notification
```

command 저장소 구현체는 `*RepositoryImpl`, query 조회 구현체는 `*FinderAdapter`, 외부 전송 구현체는 `FcmSender`처럼 이름에서 역할이 드러나게 했습니다. core는 “저장한다”, “조회한다”, “알림을 보낸다”, “공연 데이터를 동기화한다”는 port만 알고 실제 기술은 support가 책임집니다.

## 결과

| 항목 | AS-IS | TO-BE |
|---|---|---|
| usecase 가독성 | Service 안에 절차가 길어질 위험 | application Service가 흐름을 보여주고 implementation 행위자가 세부 절차 담당 |
| 행위 단위 | 조회, 생성, 저장, 참여 처리 책임이 섞일 수 있음 | `Reader`, `Writer`, `Register`, `Joiner` 등 이름으로 행위 분리 |
| domain 책임 | 업무 규칙이 Service로 흩어질 위험 | 상태 변경과 불변식은 domain 객체가 담당 |
| 구현체 의존성 | JPA, Redis, FCM, OAuth 코드가 업무 흐름으로 번질 위험 | support adapter에 구현체 격리 |
| 모듈 의존성 | core가 기술 모듈을 알게 될 위험 | core는 port만 알고 support가 구현 |

구조를 나눈 뒤 Service는 사용자 요청이 어떤 순서로 처리되는지 보여주는 역할에 가까워졌습니다. 세부 구현은 이름 있는 행위자와 adapter로 이동했고, KOPIS mapping, FCM 전송 방식, DB 조회 최적화가 바뀌어도 core usecase는 port 기준으로 유지할 수 있게 됐습니다.

## 남은 과제

모듈과 패키지가 늘어나면 처음 보는 사람이 어디에 무엇을 둬야 할지 헷갈릴 수 있습니다. 그래서 모든 기능을 무조건 세분화하기보다, usecase 흐름을 읽기 어렵게 만드는 절차와 기술 구현체부터 분리하는 기준이 필요합니다.

앞으로는 Service가 다시 많은 절차를 직접 가지지 않는지, domain 규칙이 바깥으로 새지 않는지, core가 support 구현체를 직접 알게 되지 않는지 계속 점검해야 합니다.
